# 开发规范

## 命名约定

- 类名保持包路径与文件路径一致，例如 `PHA.FACE.TPS.SA.HIS2SA.Target.OP.ConsumeTarget` 对应 `PHA/FACE/TPS/SA/HIS2SA/Target/OP/ConsumeTarget.cls`。
- 业务入口类放在 `BusinessEntry`，推送目标类放在 `Target`，数据查询类放在 `Data`。
- 方法名优先表达业务动作，避免新增临时名称；已有 `Tesst` 等历史测试方法暂不重命名，后续单独清理。

## 分层约束

- `BusinessEntry` 只负责入口信息、注册关系和业务数据获取，不直接拼接平台报文。
- `Target` 负责 payload 构建、响应解析、成功判断和结果消息。
- `Data` 负责查询 HIS 数据，不调用外部平台。
- `Platforms.Adapter` 只处理平台调用签名、入参和响应预处理。
- `BaseBizPush` 框架层不得写入 SA 专用业务逻辑。
- 新增推送编排能力优先放入 `Engine.Execute`、`Engine.Factory`、`Engine.Resolve`，避免继续膨胀 `BasePushTemplate`。
- 日志、字典、类实例化等基础设施访问应通过接口或工厂收敛，不直接散落在业务入口和 Target 中。

## ObjectScript 风格

- 新增代码优先使用现有工程风格，保持 `Set`、`Do`、`Quit` 等写法一致。
- 中文注释应说明业务约束、数据来源或平台约定，避免重复描述代码本身。
- 涉及动态对象时，字段名应与目标系统或前端协议一致，避免随意扩展。
- 出现错误状态时，优先返回结构化结果或 `%Status`，不要只写日志后静默吞掉。

## 测试约定

- 临时测试方法应标注调用示例，并尽量集中在 `Test` 类或具体 Target 的测试区。
- 与外部平台交互的测试默认不能作为自动化验收依据。
- 修改 `Data` 查询逻辑时，至少准备一组门诊或住院业务 ID 做导入后人工验证。
- 修改 `Target` 逻辑时，应验证 payload、平台返回解析和失败消息三类场景。
- 修改 `Engine.Execute` 编排时，应覆盖自动推送、人工重推、过滤、异常、平台适配器兜底五类场景。

## TODO 管理

当前仓库保留历史 TODO 和测试方法。后续清理时建议单独建任务处理：

- 确认 TODO 是否仍适用。
- 将可复用测试方法迁移到 `PHA.COM.BaseBizPush.Test`。
- 删除无效测试入口前，先确认现场是否有人依赖。
